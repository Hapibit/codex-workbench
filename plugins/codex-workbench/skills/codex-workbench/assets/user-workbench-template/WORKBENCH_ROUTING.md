# 工具和 Skill 路由

本文件用于决定什么时候调用专业 skill、MCP 或外部资料。它不记录具体项目事实。

## 基本规则

- 先判断任务类型，再选择最小足够的工具。
- 匹配 skill 时先读取 `SKILL.md`，再执行。
- optional skill 不存在时说明降级，不把缺失 skill 当成已具备能力。
- 需要当前资料、官方资料、全网比较或长期决策时，先搜索再回答。
- 对用户有成本、时间、账号、合规或安全影响的建议，必须基于资料或明确说明假设。

## 常见路由

| 任务 | 优先动作 |
| --- | --- |
| 项目工作台、项目规则、质量门、功能工作包 | 使用 `codex-workbench`。 |
| 需求不清、项目初始想法模糊、方向变化 | 先澄清，再做项目 intake。 |
| UI、Figma、设计稿、前端视觉还原 | 使用 UI/Figma 类 skill；需要真实设计资料时先取 Figma 上下文。 |
| ER 图、流程图、架构图、UML、时序图 | 使用 diagram/draw.io 类 skill。 |
| 单元测试、接口测试、Playwright、AI 对话测试 | 使用测试类 skill，并把验收证据写回项目工作台。 |
| Jenkins、GitHub Actions、CI/CD | 使用 CI/Jenkins 类 skill，并要求脚本可运行、可审查。 |
| README、技术文档、Word、论文、PPT | 使用文档类 skill，交付前做读者测试。 |
| RAG、Agent、LLM eval、安全治理 | 使用企业 AI、eval、OWASP/NIST 类 skill。 |
| 架构规划、模块边界、技术选型 | 使用架构类 skill，并输出可执行计划和验证点。 |

## 搜索规则

- “最新、官方、价格、法规、安全、API、工具用法、别人怎么做、大神怎么做、全球资料”必须搜索。
- 优先官方文档、一手仓库、标准组织、成熟工程团队资料。
- 多来源结论要区分事实、观点和推断。
- 搜索失败时说明失败原因和降级方式。

## 输出要求

- 不要求用户记 skill 名称；用户只说目标时，由 Codex 选择路由。
- 如果任务会修改项目，先说明将读取哪些项目规则。
- 如果任务只是在配置工作台，不自动推进业务项目。
