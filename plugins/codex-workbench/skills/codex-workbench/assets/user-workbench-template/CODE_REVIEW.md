# 代码审查规则

本文件用于 review、审查、复查 diff/PR/代码质量场景。审查时先读项目自己的 `REVIEW.md`。

## 审查立场

- 先报问题，再给摘要。
- 关注行为回归、安全、权限、数据、性能、可靠性、测试缺口和维护风险。
- 不把风格偏好当成高严重问题。
- 发现没有阻塞问题时明确说明，并列出剩余验证缺口。

## 严重级别

- `P0`：数据泄露、权限绕过、生产故障、不可逆数据破坏、无法回滚。
- `P1`：明确功能 bug、安全缺口、契约破坏、缺少授权检查、核心流程失败。
- `P2`：边界 bug、可维护性风险、测试薄弱、性能退化、可观测性不足。
- `Nit`：低风险可读性或命名问题，只在有实际价值时提出。

## 必查项

- 是否满足真实需求，而不是只覆盖 happy path。
- 角色、资源、租户、组织、课程、用户或其他所有权边界是否保留。
- API、数据库 schema、前后端契约、消息结构是否兼容。
- 测试或手工检查是否足以在行为坏掉时失败。
- secrets、token、cookie、日志、上传、外部调用是否安全。
- 是否避免无关重写、格式化噪音和回退用户改动。

## 输出格式

按严重级别排序列出发现：

```text
P1 - 标题
文件: path/to/file:line
风险: ...
触发场景: ...
修复方向: ...
```

没有发现阻塞问题时：

```text
未发现 P0/P1 问题。
剩余风险/验证缺口: ...
```
