# 代码质量规则

本文件用于代码实现、测试、验证、AI/RAG、UI 和文档同步场景。项目自己的规则优先级高于本文件。

## 实现前

- 先读项目规则、技术栈、现有脚本和相关代码。
- 不清楚目标、范围、验收、权限、数据或环境时先澄清。
- 优先复用已有框架、组件、工具函数、测试方式和目录结构。
- 只做当前任务需要的改动，不顺手重构无关代码。

## 实现中

- 小步修改，每一步都有可解释的目的。
- 不吞异常，不隐藏失败，不用假的成功状态代替真实验证。
- 不把临时调试、私钥、token、账号、cookie、个人路径提交进项目。
- 对权限、租户、用户数据、支付、生产数据、AI 自动写入等高风险逻辑要显式检查边界。
- 前端变更要检查布局、响应式、可访问性、文本溢出和真实交互。
- AI/RAG/Agent 变更要检查输入输出边界、工具调用权限、注入风险、引用来源和 eval 证据。

## 验证

- 小改动运行最小可靠验证。
- 跨模块、权限、数据、API、AI 输出、CI/CD 或生产影响的改动，需要扩大验证。
- 项目有质量门时，优先运行项目质量门。
- 无法运行验证时，说明原因、风险和建议的人工验证方式。

## 证据

重要功能的证据应放在项目工作台中，例如：

```text
workbench/features/<feature-name>/VERIFY.md
workbench/features/<feature-name>/REVIEW.md
workbench/features/<feature-name>/DECISIONS.md
```

重复失败不能只写聊天记录，应沉淀到测试、模板、质量门、CI、hook 或项目文档中。
