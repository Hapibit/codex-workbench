# {{PROJECT_NAME}} 证据审计规则

本文件定义项目工作台的可执行证据审计口径。分数只用于观察“证据成熟度”和“流程一致性”，不能替代产品负责人、架构师、测试、安全审查或真实用户验收的业务判断。

## 基本原则

- 评分对象是证据，不是 AI 自信程度。
- 质量门必须使用可检查条件，不能只靠 Markdown 承诺。
- AI/Agent 质量要同时看最终结果、过程证据、工具调用、数据来源和人工确认。
- 风险按影响面、可逆性、权限/数据敏感度和需求不确定性分级。
- 指标一旦变成目标就可能被刷分；分数只能作为改进信号，不能作为绕过阻塞项或人工复核的理由。
- 评分口径必须能被复核：要有明确 rubric、锚定样例、人工抽查、误报/漏报和阈值调整记录。

## 决策、总分和等级

`workbench/scorecard/scorecard.py` 输出 `decision` 和 0-100 参考分。

看结果的顺序必须是：

```text
decision
-> blockers
-> confidence
-> component floors
-> reference score
```

不要只看总分。

| decision | 含义 |
| --- | --- |
| `BLOCKED` | 存在硬阻塞，不能靠总分通过。 |
| `PASS_WITH_RISK` | 没有硬阻塞，但存在低可信度、参考分不足、组件短板、校准不足或语义/架构复核缺口，需要人工确认风险。 |
| `PASS` | 没有硬阻塞，且可信度较高；仍不能替代真实测试、审查和业务验收。 |

| 分数 | 等级 | 含义 |
| --- | --- | --- |
| 90-100 | release-ready | 证据较充分，可以进入发布或主干合并前的最终复核。 |
| 75-89 | usable-with-known-risks | 可继续开发或试用，但必须明确剩余风险。 |
| 60-74 | prototype-only | 只能作为原型或探索，不能当成可交付产品。 |
| 0-59 | not-ready | 需求、设计、验证或审查证据不足。 |
| 任意分数 + 硬阻塞 | blocked | 存在阻塞项，不能靠总分通过。 |

## 默认权重

默认权重不是统计结论，也不是永久标准。它是未校准前的启动权重，用来让项目第一次安装工作台后有一个可执行的证据审计口径。

| 维度 | 权重 | 主要证据 |
| --- | ---: | --- |
| 项目预处理 | 15 | `PROJECT_INTAKE.md` 是否 confirmed、是否有 open 阻塞问题。 |
| 产品需求 | 15 | `PRODUCT_BRIEF.md`、`PRD.md`、`ROADMAP.md` 状态和需求证据。 |
| 交互/原型 | 10 | `UX_SPEC.md`、`PROTOTYPE.md`、`USER_FLOW.md` 状态和失败/权限/可访问性证据。 |
| 架构设计 | 15 | `ARCHITECTURE.md`、`DATA_MODEL.md`、`API_DESIGN.md`、`AI_DESIGN.md` 状态、质量属性、风险和验证方式。 |
| 交付计划 | 10 | `RELEASE_PLAN.md`、`ITERATION_PLAN.md`、`TASK_BREAKDOWN.md` 状态、回滚和验证。 |
| 功能工作包 | 20 | `workbench/features/<feature-name>/` 的完整性、状态、任务、阶段门禁。 |
| 验证硬门禁 | 10 | 历史 `.workbench-validation/quality-gate-ok.json`、当前质量门上下文和质量门是否调用 scorecard。 |
| 反馈闭环 | 5 | `FAILURE_LOG.md`、`ITERATION_LOG.md`、`AI_EFFECTIVENESS.md` 是否记录返工、质量门失败和改进动作。 |

## 权重为什么这样分配

设计原则：

1. 先保证方向正确：项目预处理、产品需求、架构设计各占 15。
2. 真实开发证据最高：功能工作包占 20。
3. UX 和交付计划重要，但项目差异较大：各占 10。
4. 验证失败必须走硬阻塞，所以验证硬门禁只占 10，不用软分数表达全部重要性。
5. 反馈闭环用于长期改进，只占 5，避免用“复盘写得好”掩盖当前需求、设计或验证缺口。

具体解释：

| 维度 | 为什么是这个权重 |
| --- | --- |
| 项目预处理 15 | 项目目标、目标用户、第一版范围、权限、数据和 AI 边界错了，后续实现越多返工越大。它不能太低；但它只是方向证据，不能压过真实功能开发证据。 |
| 产品需求 15 | PRD、用户故事、非目标和验收标准决定功能是否做对。它和项目预处理、架构设计同级，因为三者共同决定“做什么”和“边界在哪”。 |
| 交互/原型 10 | 用户路径、页面状态和错误反馈对产品很重要，但后端、CLI、基础设施项目对 UX 证据要求不同，所以默认低于需求和架构。UI 密集项目应在 `CALIBRATION.md` 中调高。 |
| 架构设计 15 | 模块边界、数据流、API、AI 工具调用和权限边界一旦错，会带来高返工和高风险。默认与项目预处理、产品需求同级。 |
| 交付计划 10 | 发布范围、任务拆分、验证和回滚重要，但它更多是交付控制证据，不应压过需求、架构和功能包。 |
| 功能工作包 20 | AI 真实开发发生在功能包里。这里必须追踪规格、澄清、设计、计划、任务、实现、验证和审查，所以权重最高。 |
| 验证硬门禁 10 | 验证重要性主要由质量门和 blocker 表达。验证失败应直接 `BLOCKED`，不能靠其他维度补分。 |
| 反馈闭环 5 | 反馈闭环是长期学习机制，不能用来弥补当前 PRD、架构、功能验证不足。 |

## 为什么不是平均分

平均分默认所有维度同等重要，但真实开发风险不是平均分布：

- 功能工作包是 AI 实际动手的位置，缺失时风险最高。
- 项目预处理、产品需求和架构设计决定方向和边界，错了会产生系统性返工。
- 验证如果失败，应直接成为 blocker，而不是只扣一点分。
- 反馈闭环面向下一轮改进，不能抵消当前项目证据缺口。

所以本规则采用“加权参考分 + 硬阻塞 + 组件下限 + 校准记录”的组合，而不是平均分。

## 为什么验证只占 10 但仍然是硬门禁

验证硬门禁只占 10，不是因为验证不重要，而是因为验证失败不应该靠分数处理。

验证失败时应该直接进入：

```text
BLOCKED
-> 修复验证失败
-> 重新运行质量门
-> 更新 VERIFY.md / REVIEW.md
```

如果把验证权重调到 30 或 40，反而可能让人以为验证只是一个大权重软指标，仍然允许其他维度补分。本工作台故意把验证拆成两层：

- 质量门和 blocker：决定能不能继续。
- 参考分 10：表示验证证据成熟度。

## 档位参考线

| 档位 | 参考分 | 组件参考下限 | 校准要求 | 语义/架构复核 | 用途 |
| --- | ---: | ---: | --- | --- | --- |
| smoke | 60 | 20% | 不强制 | 不强制 | 快速检查，判断能否继续推进。 |
| standard | 75 | 50% | 不强制，但影响可信度 | 不强制，但影响可信度 | 日常开发默认证据审计。 |
| full | 85 | 60% | 必须完成 | 必须通过或明确接受风险 | 发布、PR、跨模块或高风险改动。 |

参考分和组件参考下限用于发现证据短板，不单独作为质量裁判。硬阻塞、失败测试、失败质量门、P0/P1 审查结论和未接受的高风险项才是阻断交付的依据。

## 可信度

`scorecard.py` 除了输出分数，还必须输出 `confidence`：

- `high`：没有硬阻塞、组件下限缺口、校准缺口或语义/架构复核缺口。
- `medium`：分数可作为参考，但校准、复核或警告项仍需人工看一眼。
- `low`：存在硬阻塞、组件参考下限缺口、校准不足、复核未完成或警告过多。此时总分不能作为通过依据。

可信度比总分更重要。低可信度高分只能说明文档结构看起来完整，不能说明项目真的健康。

## 校准

评分校准记录在 `CALIBRATION.md`。它必须回答：

- 这个分数到底测什么、不测什么。
- 哪些锚定样例应该低分、中分、高分。
- 人工抽查和脚本结论是否一致。
- 出现过哪些误报、漏报。
- 参考线为什么调整。

推荐校准步骤：

1. 选 3-5 个真实功能包作为锚定样例。
2. 人工或独立审查先判断能不能交付、风险在哪里、缺什么证据。
3. 运行 scorecard，对比脚本结论。
4. 记录 false positive 和 false negative。
5. 根据误判调整模板、硬阻塞、组件下限、权重或脚本。
6. 再跑一次验证，记录调整原因。

没有校准时，`smoke` 和 `standard` 可以继续作为过程提示，但不能把高分当成质量证明；`full` 档必须产生硬阻塞，除非明确接受风险。

## 硬阻塞

出现以下情况时，不管总分多少都不能通过：

- `PROJECT_INTAKE.md` 有 open 阻塞问题。
- 质量门没有调用 `workbench/scorecard/scorecard.py` 生成证据审计报告。
- `SCORECARD.md` 结构缺失，无法记录语义复核和架构复核。
- 功能工作包缺文件、状态非法、阶段顺序倒置或 complete 后仍有未勾选任务。
- `current_stage` 已进入后续阶段，但前置 SPEC、CLARIFY、DESIGN、PLAN、TASKS、VERIFY 或 REVIEW 没有对应通过证据。
- 高风险权限、数据、生产、AI 自动生效变更缺少验证或人工确认。

## 人工/AI 语义复核

脚本只能检查证据是否存在、状态是否一致、质量门是否执行。以下内容必须由人或独立 AI 审查补充到 `SCORECARD.md`：

- 产品目标是否真的合理。
- PRD 是否覆盖真实用户场景。
- UX 是否符合目标用户工作流。
- 架构是否适合项目规模、团队能力、数据边界和未来扩展。
- AI/RAG/Agent eval 是否覆盖真实失败样例。
- 安全、隐私、权限和合规判断是否可信。

语义复核状态只能使用：

- `passed`
- `accepted_with_risk`
- `not_required`
- `pending`
- `failed`

`pending` 或 `failed` 不一定让脚本硬失败，但必须在交付说明里作为剩余风险说明。高风险项目应把它提升为人工确认门禁。

`full` 档会把语义复核和架构复核提升为硬门禁。原因是发布、PR、跨模块和高风险改动不能只依赖脚本检查“有没有填文档”。

## 防刷分规则

- 不得为了通过分数而补空表、假状态或没有验证证据的段落。
- 分数下降不等于失败，可能只是暴露了真实风险。
- 分数上升必须能对应到更清晰的需求、设计、测试、审查、质量门或自动化。
- 发现 scorecard 误判时，不要手改结果；记录到 `CALIBRATION.md`，再调整模板、脚本、参考线或审查规则。
- 重复失败要进入 `workbench/feedback/FAILURE_LOG.md`，能自动化的进入测试、质量门、CI 或 hook。
