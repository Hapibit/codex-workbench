每个任务必须小、可执行、可验证。不要把多个风险不同的改动塞进一个任务。

任务写法：

```text
- [ ] N. 任务标题。
  - 依据：SPEC/PLAN 中的哪一项。
  - 输入：需要的文件、数据、配置或确认。
  - 输出：完成后应该出现的文件、行为或证据。
  - 预计文件：
  - 验证：
  - 回滚：
```

## 任务拆分规则

- 一个任务应该能独立解释为什么要做。
- 一个任务应该能说明完成后怎么验证。
- 一个任务应该有明确输入、输出和证据位置。
- 一个任务失败时不应该让整个项目处于不可理解状态。
- 高风险任务前必须有确认点。
- 涉及数据库、权限、公开 API、AI 自动生效或生产发布的任务，不能和低风险 UI/文档任务混在一起。
- 如果任务执行中发现 SPEC、DESIGN 或 PLAN 不对，先停止并同步更新上游文档。
